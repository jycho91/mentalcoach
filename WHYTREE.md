# WHYTREE: 문제 탐색 → 솔루션 도출

## ⬆️ Why Up (문제 파기)

**출발점**: "기업의 규정 관리자는 항상 바쁘다"

**Why 1**: 왜 항상 바빠?
→ 외부 법령이 수시로 개정되는데, 그때마다 사내 규정을 맞춰서 바꿔야 하니까.

**Why 2**: 왜 규정 바꾸는 게 그렇게 부담이야?
→ 규정 하나 개정하려면 개정 사유서, 주요 골자, 신구조문대비표를 전부 수작업으로 만들어야 해서. 문서 작업이 본업을 잡아먹음.

**Why 3**: 왜 자동화가 안 돼 있어?
→ "법 조문이 바뀌면 우리 규정 어디가 영향받는가"를 판단하는 건 의미를 이해해야 하는 일이라, 키워드 검색이나 규칙 기반으론 불가능했음.

**Why 4**: 왜 방치하면 안 돼?
→ 법령이랑 사내 규정이 어긋나면 행정처분·벌금·소송. 중대재해처벌법 이후로 경영진 개인 책임까지 확대됨. 컴플라이언스 미스 = 경영 리스크.

**Why 5**: 왜 기존 서비스로 안 풀려?
→ 로앤비·케이스노트는 법령 "검색"만 됨. 사내 규정이랑 연결 안 됨. 더존 같은 규정 관리 툴은 법령 변화를 모름. 둘을 잇는 게 없음.

**도달한 핵심 문제**
> "법령 변화 → 사내 규정 영향 분석 → 개정안 작성, 이 파이프라인이 통째로 수작업이고, 이걸 자동화할 기술이 이제야 가능해졌다."

---

## 🔀 전환점: 그래서 뭘 만들어야 해?

LLM + 임베딩 + MCP(실시간 공공 데이터 연동) 기술이 성숙한 지금, 이 파이프라인을 통째로 자동화하는 B2B SaaS를 만들 수 있다.

→ **레귤메이트(RegulMate) 도출**

## ⬇️ How Down (솔루션 구체화)

**How 1**: 법 바뀐 걸 어떻게 실시간으로 알아?
→ Korea Law MCP로 국가법령정보센터 실시간 폴링 → 📡 Step 1

**How 2**: 우리 규정 중 뭐가 영향받는지 어떻게 찾아?
→ 사내 규정 10개 임베딩 + 개정 조문 의미 매칭 → 📡 Step 1

**How 3**: 왜 바꿔야 하는지 어떻게 설명해?
→ 디텍팅된 규정마다 개정 필요 사유 + 근거 법 조문을 함께 표시 → 📡 Step 1

**How 4**: 개정안 문서를 어떻게 자동으로 만들어?
→ 사용자가 대상 규정 선택 → LLM이 개정 사유 + 주요 골자 + 신구조문대비표 생성 → 📝 Step 2

**How 5**: 법적으로 괜찮은지 어떻게 검증해?
→ 대법원 판례 DB RAG 교차 검증 → 🛡️ 기능 2

**How 6**: 부서별 규정끼리 모순은?
→ 규정 간 임베딩 + 허용/금지 패턴 탐지 → ⚖️ 기능 4

**How 7**: 실무자가 애매한 상황에서 바로 물어보려면?
→ 사내 규정 + 외부 판례 통합 챗봇 → 🤖 기능 5

## 🌳 전체 흐름

```
[관찰] 규정 관리자는 항상 바쁘다
   ↓ why
[원인] 법령이 바뀔 때마다 규정을 맞춰 바꿔야 한다
   ↓ why
[병목] 개정 사유서·신구조문대비표가 전부 수작업
   ↓ why
[근본] 법령↔규정 의미 매칭이 기술적으로 불가능했다
   ↓ why
[공백] 기존 서비스가 이 파이프라인을 안 다룬다
   ↓ 전환점: LLM+MCP 기술 성숙
[솔루션] 레귤메이트
   ↓ how
[Step 1] 디텍팅: 스캔 → 영향 규정 + 사유/근거
   ↓ 사용자 선택
[Step 2] 개정안 생성: 신구조문대비표 출력
   ↓ how
[기능 2~5 확장]
```
